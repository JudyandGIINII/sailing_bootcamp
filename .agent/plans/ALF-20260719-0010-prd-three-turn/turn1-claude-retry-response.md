## 1. 총평

L01–L05·Training Sloop 단일 프로필·결정론적 재현이라는 스코프 자체는 명확하지만, 그 스코프를 실제로 검증 가능하게 만드는 계약 층위(단위/좌표계, 결정론의 정의, 레슨별 정량 회복 기준, 루브릭 가중치, 릴리스 게이트 임계값)가 비어 있어 지금 상태로는 개발·QA·파일럿 어느 단계에서도 "완료"를 객관적으로 판정할 수 없다.

## 2. severity-ranked findings

**[Critical] 8.2 / FR-01 / NFR "결정론·회귀성" — 결정론 계약의 범위가 정의되지 않음**
"`scenario version + seed + input log + model version + boat profile version`이 같으면 같은 **주요 상태**·채점 결과"라고만 명시되어 있고, "주요 상태"의 경계, 부동소수점/브라우저·GPU 간 재현성 보장 범위, input log가 실시간(wall-clock) 기준인지 시뮬레이션 틱 기준인지가 정의되어 있지 않다. 이는 golden replay 회귀 테스트(NFR)와 FR-08 재도전 비교 기능이 무엇을 "일치"로 판정할지 테스트 설계 자체를 불가능하게 만든다.
→ 추가 제안: "주요 상태"를 물리적으로 열거(위치, heading, STW/SOG, 수심여유, 이벤트 로그, 점수)하고, input log는 고정 tick 기반 이산 스텝으로 기록하며, 부동소수점 재현성은 단일 결정론적 물리 스텝 함수(플랫폼 독립)로 계산한다는 문장을 8.2에 추가.

**[Critical] 8.1 / 8.2 / FR-04 — 시뮬레이션 단위·좌표계·부호 규약 부재**
바람/조류/heading/COG/조위/흘수 어디에도 단위(m/s vs kn), 각도 기준(진북 vs 자북, 시계방향 정의), 조류 벡터 부호 규약이 없다. FR-04 수용 기준("벡터 합성과 맞는다")은 좌표계 없이는 테스트 불가능하다.
→ 추가 제안: 8.1/8.2에 "단위·좌표계" 서브섹션 신설 — 속도는 knots, 각도는 진북 기준 0–360° 시계방향, 수심/흘수는 m, apparent wind는 boat-relative 각도로 표준화한다고 명시.

**[Critical] 7.2 표 / FR-06 — 레슨별 정량 안전 회복 기준이 실제로 없음 (FR-06과 내용 불일치)**
FR-06 수용 기준은 "각 MVP 레슨에 통과·실패·안전 회복 조건이 정의된다"고 약속하지만, 7.2 표는 "위험 전에 세일 축소/안전 조타 선택"(L03) 같은 정성적 목표만 있고 트리거 임계값(돌풍 풍속, heel 각도, 반응 시간 창)이 전혀 없다. 이는 채점 결정론(8.2)과도 충돌한다 — 임계값 없이는 "위험 이벤트"를 결정론적으로 발생시킬 수 없다.
→ 추가 제안: L01–L05 표에 "안전 회복 조건" 열을 신설하고, 각 레슨에 최소 1개의 정량 임계값(예: L03 = 돌풍 풍속 X kn 진입 후 Y초 내 reef/침로 변경 시 안전 회복 인정)을 명시. 실제 수치는 도메인 전문가 검증 필요(§3 참조).

**[High] 7.3 — 채점 루브릭에 가중치/공식이 없음**
"관측, 판단, 조작 안정성, 안전, 목표 달성" 5개 범주만 나열되고 상대 가중치, 감점/가점 공식, "안전 위반이 속도 이득을 상쇄하지 않는다"의 구체적 계산 방식이 없다. 제품 원칙 2("설명 가능성이 사실성보다 먼저")와 NFR "설명 가능성"(숨은 규칙 금지)에 정면으로 배치되는 공백이다.
→ 추가 제안: 7.3에 "총점 = f(관측 이벤트, 판단 이벤트, 조작 안정성, 안전 위반 수, 목표 달성)"의 최소 구조와 "안전 위반 1회당 목표달성 점수 상한 적용" 같은 하드 룰을 명시.

**[High] 6.1 / D1 / 10 — 입력 지연(latency)·프레임 예산이 전면 후속 결정으로 미뤄짐**
D1은 "정확한 지원 브라우저·성능 예산은 기술 옵션 비교에서 확정"이라 하고, §12 출시 전 의존성에도 동일 항목이 있다. 그러나 이 제품은 helm/sheet를 실시간 조작하는 게임이며, L03(돌풍 반응)처럼 반응 시간 자체가 학습·채점 대상이다. 입력 지연이 정의되지 않으면 "안전 회복 조건"(FR-06)과 결정론(8.2) 둘 다 흔들린다. 완전 후속으로 미루는 것은 접근성(운동 제약 사용자)에도 영향을 준다.
→ 추가 제안: MVP 최소 기준으로 "입력→시뮬레이션 반영 지연 ≤ 1 tick(또는 Xms), 목표 프레임레이트 Y fps"를 10절에 잠정치로 명시하고, 세부 값만 기술 옵션 비교로 위임.

**[High] §10 수용기준 8·9 / §12 — 파일럿·전문가 검토 게이트에 통과 임계값이 없음**
"도메인 전문가가 검토한다", "파일럿에서 사전/사후 이해도를 측정한다"는 활동만 정의되어 있고, 몇 명 대상인지, 어떤 이해도 점수/전문가 승인 기준이 있어야 출시 가능한지(go/no-go 기준)가 없다. 이는 리스크 표의 "잘못된 자신감 유발" 완화책("도메인 검토")을 형식적 체크박스로 만들 위험이 있다.
→ 추가 제안: 수용기준 8·9를 "전문가 N인 이상 서면 승인 + 핵심 인과관계(예: L04 drift 보정) 이의 없음", "파일럿 참가자 N명 이상, 사후 이해도 사전 대비 X%p 이상 향상 시 통과"처럼 정량화.

**[Medium] §10 — 접근성 요구가 최소 수준으로만 정의됨**
"키보드 조작, 색상 외 구분, 일시정지/속도조절"만 있고 WCAG 목표 등급, 포커스 순서/스크린리더 지원 여부(또는 명시적 제외), 색맹 대비비 검증 방법, 청각 신호(경보음 등) 존재 시 대체 표시가 없다. FR-02 HUD가 시정/경보 상태를 표시한다고 했는데 이것이 시각 전용인지 불명확하다.
→ 추가 제안: "WCAG 2.1 AA 중 키보드/색상대비 항목 준수, 스크린리더 지원은 MVP 제외를 명시적으로 선언"처럼 범위를 확정해 §10에 추가.

**[Medium] FR-10 / D5 — 범용 카탈로그 스키마 요구가 "Sloop 단일" MVP 원칙과 긴장 관계**
D5는 "MVP는 Sloop 하나로 고정"이라 하는데, FR-10은 monohull/multihull, mast count/role/position, sail inventory를 "별도 필드로" 버전 관리하는 범용 카탈로그 구축을 MVP 요구사항으로 요구한다. 플레이 가능한 프로필이 1개뿐인 상태에서 멀티헐/다중 마스트까지 아우르는 일반화된 데이터 모델을 미리 구축하는 것은 리스크 표 1행("MVP 범위 폭발")이 우려하는 것과 같은 종류의 선행 투자이며, 어떤 레슨도 이를 요구하지 않는다.
→ 추가 제안: FR-10을 "MVP는 `training-sloop-v1` 단일 레코드만 채우되, 스키마는 후속 확장 필드를 예약(reserved)한다" 수준으로 낮추거나, 범용 스키마 구축이 MVP 일정에 포함되는 이유를 명시.

## 3. UNVERIFIED — 도메인 검증 필요

- **수심 여유 공식 `수심 + 조위 - 흘수`(8.1):** heel로 인한 흘수 증가, squat, 조석 기준면(datum) 종류(MLLW 등)를 반영하지 않음 — 안전 교육 콘텐츠로서 오해를 유발하지 않는지 세일링 강사 검증 필요.
- **L03 돌풍/파도/reef의 안전 반응 창(reaction window):** 몇 kn 돌풍에서 몇 초 내 reef/감속이 "안전"으로 간주되는지는 실제 교육 기준(RYA/US Sailing 등)에 근거해야 하며, 본 PRD·리서치 문서 어디에도 수치 근거가 없음(§3.1 §3.2 참조).
- **조류 단순화(수직 수층·복합 해류 제외, 8.1):** STW/SOG/COG 인과관계 학습에 이 단순화가 여전히 정확한 멘탈 모델을 주는지, 혹은 실제 조류 거동과 괴리된 학습을 유발하는지 확인 필요 — 벤치마크 문서(NOAA Currents)는 이 단순화의 교육적 타당성을 검증하지 않았다고 스스로 명시(출처 3).
- **Training Sloop 대표 폴라·흘수·안전 임계값(§12에 이미 미결정으로 기재):** 현재 PRD 어디에도 실제 수치가 없어 L01–L05 전체가 가정치 위에 설계됨 — 출시 전 의존성 그대로 유효하며 강조 재확인.
- **파도의 "선속·조타 안정성 보정"만으로 전복/승선 안전을 암시하지 않는지(6.2에서 전복/복원 명시 제외):** 파도 패널티가 사용자에게 실제 안전 여유로 오인되지 않도록 하는 문구/UX 검증이 필요.
